# 紫微戏命师美术资源接入与移动端性能方案

版本：v1.0  
适用项目：`tii-mom/Dreamer` / `xms-dialogue-game`  
核心目标：把已经生成或即将生成的美术资源完整接入项目，同时避免用户手机一次性下载、解码、缓存过多大图，保证移动端首屏速度、抽卡爽感、微信传播卡片质量和长期资源可维护性。

---

## 0. 结论

当前项目已经规划了完整首发美术资源体系，但代码层还没有真正资产化接入。下一步不能简单把大图全部丢进组件里直接 `<img src="/assets/...">`，否则会带来：

1. Git 仓库和部署包持续膨胀。
2. 移动端页面同时解码多张大图，导致内存升高、卡顿、白屏或闪退。
3. 首页、聊天、盲盒、市场、铭文背包都可能误加载非当前场景资源。
4. 后续微信机器人、经营者命铺、朋友圈分享卡无法统一复用资源。
5. 缺少资源清单、尺寸、稀有度、用途、版本和缓存策略，后期很难维护。

推荐方案：

> **源图归档，线上只使用压缩派生图；资源通过 Manifest 管理；首屏只加载关键小图；列表只用缩略图；抽中/详情才加载大卡；大批量资源优先放 R2/CDN；前端使用懒加载、响应式图片、占位图和预加载策略。**

## 2. 资源目录策略

### 2.1 推荐目录结构

建议重构为以下结构：

```text
art-source/
  masters/
  blind-box/
  runes/
  share/
  marketing/
  ui/

public/assets/
  critical/
    master-avatar-96.webp
    master-avatar-192.webp
    coin-96.webp
    ticket-96.webp

src/lib/assets/
  asset-types.ts
  asset-manifest.generated.ts
  asset-url.ts
  image-presets.ts

scripts/
  build-art-assets.mjs
  upload-art-assets.mjs
  verify-art-assets.mjs
  generate-asset-manifest.mjs
```

解释：

- `art-source/`：原始生成图归档，不直接被前端引用。可以不随生产包发布。
- `public/assets/critical/`：只放首页必须马上显示的小资源。
- 大多数盲盒、铭文、营销、分享资源建议上传 R2，通过 `/api/assets/...` 或 CDN 地址访问。
- `asset-manifest.generated.ts` 是前端唯一可信资源入口，不允许组件手写散乱路径。

### 2.2 不建议的做法

不要这样做：

```tsx
<img src="/assets/masters/renji/renji-01-card.webp" />
```

原因：

1. 无法统一切换 R2/CDN。
2. 无法知道图片宽高、字节大小、稀有度、用途。
3. 无法自动给 `srcSet`、`sizes`、placeholder。
4. 后续资源改名会全项目搜索替换。
5. 很容易在首页误加载大图。

正确方式：

```tsx
<AssetImage assetId="master.renji.01" variant="thumb" />
<AssetImage assetId="master.renji.01" variant="card" priority={isDrawResult} />
```

## 3. 图片派生规格

### 3.1 角色卡 Master Card

源图：

```text
1024x1536 或更高
```

线上派生：

| Variant |      尺寸 | 格式        | 用途               |  目标大小 |
| ------- | --------: | ----------- | ------------------ | --------: |
| `thumb` |   240x360 | WebP / AVIF | 市场列表、抽卡历史 |  <= 45 KB |
| `list`  |   320x480 | WebP / AVIF | 背包、列表大卡     |  <= 80 KB |
| `card`  |  720x1080 | WebP / AVIF | 抽中结果、详情页   | <= 180 KB |
| `full`  | 1024x1536 | WebP / AVIF | 高清预览、分享生成 | <= 350 KB |

首屏不要加载 `card/full`。

### 3.2 头像 Avatar

| Variant |    尺寸 | 用途              | 目标大小 |
| ------- | ------: | ----------------- | -------: |
| `sm`    |   64x64 | 聊天气泡          |  <= 8 KB |
| `md`    |   96x96 | 顶部栏/列表       | <= 15 KB |
| `lg`    | 192x192 | 详情/个人页       | <= 35 KB |
| `raw`   | 512x512 | 分享生成/头像详情 | <= 90 KB |

### 3.3 铭文 Runes

| Variant |    尺寸 | 用途        |
| ------- | ------: | ----------- |
| `icon`  |   96x96 | 装配槽      |
| `thumb` | 160x160 | 背包        |
| `card`  | 320x320 | 铭文详情    |
| `raw`   | 512x512 | 活动页/分享 |

### 3.4 盲盒资源

| 类型       | Variant  | 规格                    |
| ---------- | -------- | ----------------------- |
| 盲盒外观   | `box`    | 512x512                 |
| 开盒序列帧 | `frame`  | 512x512，每帧 <= 80 KB  |
| 稀有度闪光 | `effect` | 512x512 或 CSS/SVG 优先 |
| 结果背景   | `mobile` | 720x1280                |
| 结果背景   | `share`  | 1080x1920               |

开盒动效 MVP 不建议视频，建议 CSS + 6 帧 WebP + 粒子图。不要预加载全部稀有度动效，只预加载当前抽奖用到的通用帧，抽中结果确定后再加载对应稀有度光效。

### 3.5 分享与营销资源

| 类型      |      尺寸 | 用途            |
| --------- | --------: | --------------- |
| 分享背景  | 1080x1920 | 微信朋友圈/长图 |
| H5 展示版 |  720x1280 | 页面预览        |
| 海报      | 1080x1920 | 营销页          |
| 缩略海报  |   360x640 | 列表/后台选择   |

分享图片最终可以由服务端 SVG/HTML 合成，也可以前端 Canvas 合成。底图应只加载当前用户要生成的那一张。

## 4. 格式策略

### 4.1 推荐格式

| 场景      | 首选                | 备选    |
| --------- | ------------------- | ------- |
| 角色卡    | AVIF                | WebP    |
| 头像      | WebP                | PNG     |
| 铭文      | WebP                | PNG/SVG |
| 徽章/边框 | SVG                 | WebP    |
| 分享背景  | WebP                | AVIF    |
| 透明立绘  | WebP lossless / PNG | PNG     |
| UI 小图标 | SVG/WebP            | PNG     |

### 4.2 为什么不要只用 PNG

PNG 适合透明图和小图标，但角色卡、背景、海报用 PNG 会非常大。用户手机真正的压力不是文件存储，而是下载、解码和渲染内存。大 PNG 一旦解码，会按像素展开成 RGBA，占用内存远高于文件大小。

示例：

```text
1024 x 1536 x 4 bytes ≈ 6 MB 解码内存
1080 x 1920 x 4 bytes ≈ 8 MB 解码内存
```

如果一个页面同时渲染 10 张大图，就会非常危险。

## 5. 存储与分发策略

### 5.1 两层资源分发

#### 第一层：`public/assets/critical`

只放少量关键资源：

```text
master-avatar-96.webp
master-avatar-192.webp
coin-96.webp
ticket-96.webp
loading-oracle-160.webp
```

这些会随前端静态资源部署，保证首屏稳定。

#### 第二层：R2 / CDN 资源

大部分图片上传到 R2：

```text
/api/assets/art/masters/renji/renji-01-card-720.webp
/api/assets/art/masters/renji/renji-01-thumb-240.webp
/api/assets/art/runes/silver/S01-icon-96.webp
/api/assets/art/blind-box/results/result-bg-renji-720.webp
```

当前项目已经有 R2 bucket 配置和 `/api/assets/` 读取逻辑，可以在此基础上扩展。

### 5.2 Cache-Control

所有带 hash 的资源使用：

```http
Cache-Control: public, max-age=31536000, immutable
ETag: <r2-etag>
```

如果资源会替换，不要覆盖同名文件，使用 hash 文件名：

```text
renji-01-card-720.ab12cd.webp
```

Manifest 里引用 hash 文件。

### 5.3 不要让 Service Worker 缓存所有大图

PWA 可以缓存 critical 资源，但不要把 141 个资源一次性 precache。建议：

- Precache：critical 图标、首屏 avatar、loading 图。
- Runtime cache：用户实际看过的卡图、铭文图。
- Cache limit：最多 50～80 张，LRU 淘汰。
- 单资源过大不缓存，或者只缓存缩略图。
- 分享海报、营销海报不默认缓存。

## 7. 前端加载策略

### 7.1 统一组件 `AssetImage`

新增：

```text
src/components/assets/AssetImage.tsx
```

功能：

- 从 manifest 查 URL。
- 自动填 `width` / `height`，避免布局抖动。
- 默认 `loading="lazy"`。
- 默认 `decoding="async"`。
- 支持 `priority`，只有首屏关键图或抽中结果主卡使用。
- 支持 blur placeholder。
- 支持失败兜底。
- 支持 `srcSet`。
- 支持 `sizes`。
- 支持容器 className。

示例：

```tsx
export function AssetImage({
  assetId,
  variant = "thumb",
  priority = false,
  alt,
  className,
}: Props) {
  const asset = getAsset(assetId);
  const file = pickVariant(asset, variant);

  return (
    <img
      src={file.url}
      width={file.width}
      height={file.height}
      alt={alt || asset.name}
      loading={priority ? "eager" : "lazy"}
      decoding={priority ? "sync" : "async"}
      className={className}
      style={{
        backgroundColor: asset.dominantColor,
      }}
    />
  );
}
```

### 7.2 场景加载规则

#### 首页 / 微信绑定页

只加载：

- 1 个 96/192 avatar
- 1 个背景或 CSS 渐变
- 不加载盲盒、铭文、市场大图

#### 聊天页

- 聊天气泡头像使用 `avatar_sm`
- 只有用户点击卡片详情才加载角色 `card`
- 聊天历史不要回放加载所有图片，已离屏的图片使用懒加载

#### 盲盒页

- 进入盲盒页：预加载盲盒盒子图、通用开盒帧
- 点击抽奖：不要立即加载所有结果卡
- 抽奖结果确定后：预加载对应结果背景 + 对应角色 card
- 市场参考价等文字先显示，图片可渐进显示

#### 市场 / 背包页

- 列表只用 `thumb`
- 滚动列表使用虚拟列表或分页
- 点击详情才加载 `card/full`
- 不要一次渲染 100 个资源卡片

#### 铭文装配页

- 装配槽用 `icon`
- 背包 grid 用 `thumb`
- 铭文详情才用 `card`

#### 分享生成页

- 页面预览用 720 版
- 生成高清分享图时，才加载 1080/1920 版
- 生成完成后释放 Canvas 和临时 Object URL

## 8. 移动端内存控制

### 8.1 基本原则

1. 同屏最多 1～2 张大图。
2. 列表永远用缩略图。
3. 大图出场要有用户动作触发。
4. 离屏资源不要保持解码状态。
5. 开盒动画结束后卸载序列帧 DOM。
6. 分享生成完成后释放 canvas。
7. Service Worker 不预缓存大图。
8. 避免 CSS 背景一次性加载大图。

### 8.2 具体实现

盲盒开盒动效：

```tsx
{
  phase === "opening" && <OpeningSequence />;
}
{
  phase === "result" && <DrawResultCard />;
}
```

不要让 `OpeningSequence` 和 `DrawResultCard` 同时常驻 DOM。

市场列表：

```tsx
const visibleItems = items.slice(page * 20, page * 20 + 20);
```

或者用虚拟列表：

```text
react-window / @tanstack/react-virtual
```

图片失败兜底：

```tsx
onError={() => setFailed(true)}
```

兜底展示 CSS 稀有度卡框 + 默认 avatar，不要空白。

## 9. 构建脚本设计

### 9.1 `scripts/build-art-assets.mjs`

输入：

```text
art-source/**/*.png
art-source/**/*.webp
```

输出：

```text
dist-art/
  art/masters/renji/renji-01-thumb-240.hash.webp
  art/masters/renji/renji-01-card-720.hash.webp
  manifest/art-manifest.generated.ts
  manifest/art-report.json
```

功能：

1. 扫描源图。
2. 校验命名规则。
3. 生成多尺寸 WebP/AVIF。
4. 生成 blur placeholder。
5. 计算 dominant color。
6. 计算 hash。
7. 写入 manifest。
8. 输出体积报告。

推荐工具：

```text
sharp
```

需要新增依赖：

```bash
npm i -D sharp
```

### 9.2 `scripts/upload-art-assets.mjs`

功能：

1. 读取 `dist-art/art/**`
2. 上传到 R2：
   ```bash
   wrangler r2 object put xms-dialogue-game-assets/art/masters/... --file ...
   ```
3. 设置 content-type。
4. 输出上传报告。
5. 不上传 source 原图。

### 9.3 `scripts/verify-art-assets.mjs`

检查：

- 所有 manifest URL 可访问。
- 所有资源有 width/height/bytes。
- 角色卡不能超过目标大小。
- 首屏 critical 总体积不能超过 300 KB。
- 盲盒首屏资源不能超过 800 KB。
- 单张 full 图不能超过 500 KB。
- 所有引用的 assetId 在 manifest 存在。
- 未引用的 public 大图给 warning。

## 12. 资源和业务数据绑定

### 12.1 不要只靠文件名

需要在业务层定义资产：

```ts
export const masterCatalog = [
  {
    code: "renji-01",
    assetId: "master.renji.01",
    name: "人玑命师 · 星座塔罗",
    rarity: "renji",
    specialty: "星座塔罗",
    tradable: true,
    defaultRuneSlots: 1,
  },
];
```

盲盒抽奖返回：

```ts
{
  itemType: "master",
  itemCode: "renji-01",
  assetId: "master.renji.01",
  rarity: "renji"
}
```

前端通过 `assetId` 找 manifest 图片。

### 12.2 铭文同理

```ts
{
  code: "S01",
  assetId: "rune.silver.S01",
```

## 13. 微信传播图片策略

### 13.1 分享卡生成方式

推荐服务端生成分享卡，而不是让前端截图。

原因：

1. 微信里分享图片需要稳定 URL。
2. 前端截图在不同手机上质量不稳定。
3. 服务端可统一加二维码、水印、经营者 referral code。
4. 生成后可上传 R2，重复分享不重复生成。

流程：

```text
用户 / 经营者请求生成分享卡
↓
服务端读取 share 背景 asset
↓
合成标题、判词、角色头像、二维码、referral code
↓
上传 R2
↓
返回 shareAsset.url
↓
微信机器人 / H5 页面发送该 URL
```

当前项目已有 `generateShareCard` 和 R2 上传逻辑，可以在此基础上升级。

### 13.2 分享卡尺寸

| 场景             |      尺寸 |
| ---------------- | --------: |
| 微信朋友圈长图   | 1080x1920 |
| 微信聊天卡片预览 |  900x1200 |
| H5 页面预览      |  720x1280 |

### 13.3 分享卡缓存

生成后的分享卡应使用：

```text
share/{userId}/{shareId}.webp
```

如果内容不可变，使用 immutable cache。

## 14. 验收标准

### 14.1 资源完整性

- Manifest 中覆盖文档要求的所有 MVP 资源。
- 资源 ID、文件路径、业务 catalog 一一对应。
- 页面不再手写散乱 `/assets/...` 路径。
- 缺图时有兜底，不影响页面。

### 14.2 性能

移动端目标：

| 指标                    |      目标 |
| ----------------------- | --------: |
| 首页初始图片请求        |      <= 5 |
| 首页图片总大小          | <= 300 KB |
| 盲盒页首次图片总大小    | <= 800 KB |
| 市场列表首屏图片总大小  |   <= 1 MB |
| 单张角色详情大图        | <= 350 KB |
| 低端 Android 抽卡不卡死 |  必须通过 |
| Lighthouse Performance  |     >= 75 |
| 无一次性加载 141 资源   |  必须通过 |

### 14.3 网络

- 图片有 `width/height`，无明显 CLS。
- 图片有 `Cache-Control`。
- R2 资源返回正确 `content-type`。
- 资源带 hash 或 manifest version。
- 离线/弱网有占位和重试。

### 14.4 业务

- 抽中角色展示真实卡图。
- 背包展示真实头像/缩略图。
- 铭文装配展示真实图标。
- 市场展示真实缩略图。
- 分享卡使用真实背景和二维码框。
- 微信机器人发送图片 URL，不发送大 base64。
